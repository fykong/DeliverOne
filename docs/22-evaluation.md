# 独立验收评测(方法、结果与改进)

对应提交材料选填项「评测方案与样例结果」。回答一个核心质疑:**"模型自己写的测试通过"不等于"输出真的正确"**——生成系统既当运动员又当裁判时,需要独立的第三方审判。本评测的价值不在于得到满分,而在于**它真的抓到了"测试全绿"掩盖的缺陷**。

## 评测方法

- **被评对象**:平台在 Conduit 实仓上真实交付的 4 条提测分支(L1 字数统计、L2 跨栈封面图、L1 列表打标、L2 时间展示)。
- **审计员独立性**:由与运行时模型(doubao-seed-2.0-lite)完全无关的另一模型体系担任,从未接触生成过程,指令明确"发现问题是价值,放水是失职"。
- **审需求而非审测试**:每任务从原始需求拆 5-7 条验收标准,审计员必须打开真实代码核对、对计数/时间逻辑人工推演具体输入输出、独立运行 `npm test -- --run`、**独立判断测试断言本身是否正确**(防"实现错了测试跟着错"的共谋)。
- **对抗复核**:FAIL 判定再派独立复核员审一遍防误杀。

## 评测结果(总分:17 pass / 3 partial / 3 fail)

| 任务 | 结论 | 核心发现 |
|---|---|---|
| L1 字数统计 | **accept-with-defects** | 计数/阅读时长经 3 组人工验算正确,18/18 测试过;链接锚文本被误删(少算)、Markdown 剥离测试判别力弱 |
| L2 时间展示 | **accept-with-defects** | 相对时间梯度与 now 注入正确,渲染接入真实 updatedAt;缺非法日期守卫、未覆盖精确边界值的测试 |
| L1 列表打标 | **reject** | isHotTag 纯函数与测试正确,但**从未被组件导入**,页面不会出现"热"徽标——需求核心未交付 |
| L2 封面图 | **reject** | 后端模型/迁移/控制器质量合格,但前端 `setArticle.js` 未同步改,coverImage 在发请求前被丢弃,UI 端到端不通 |

### 全局系统缺陷(已修复)

所有 4 条分支的 `package.json` 都被注入 `"ai-delivery-workbench-v2": "file:../../../.."`——沙盒嵌套在工作台(npm workspace)内,沙盒里 `npm install` 把工作台自身作为 file 依赖注入,导致交付物在别的机器上 `npm install` 会失败。
**修复**:提测链路 `git_submission._sanitize_dependency_pollution()` 在 commit 前把 `package.json`/`package-lock.json` 还原到基线(需求实现只新增源码、从不改依赖清单),并对无基线场景结构化剔除自引用。已重新净化 [PR #1](https://github.com/fykong/conduit-realworld-example-app/pull/1) 与 [PR #3](https://github.com/fykong/conduit-realworld-example-app/pull/3),净 diff 仅含需求源码文件。

## 关键洞察:为什么需要分层验证

两个 reject 的共同点是**"可验证的逻辑正确,但接线/渲染缺失"**——这恰好揭示了单一验证层的盲区,也印证了本系统三层验证设计的必要:

| 验证层 | 能抓什么 | 抓不到什么 |
|---|---|---|
| **测试级**(vitest 单测) | 纯函数逻辑、计数/时间正确性 | 纯函数有没有被组件**真正用上**(本次两个 reject 全部漏网) |
| **页面级**(visual gate 浏览器断言) | "热"徽标/封面图**是否真的出现在页面上** | 需预览进程运行;数据依赖页面需数据库 |
| **独立审计**(本评测) | 接线缺失、需求遗漏、测试共谋、依赖污染 | 非自动化,作为终审 |

**本次两个 reject 若在预览进程运行下执行,页面级 visual gate 的"expectedText/requiredSelector 断言"会直接判 fail 并触发修复循环**——这正是 visual gate 的设计目的。当前这两次运行未启动预览,visual gate 被如实标记 skipped,缺陷因而下沉到独立审计层才被捕获。**结论:把 visual gate 设为前端类任务的默认门禁(自动拉起 vite 预览)是首要改进项。**

## 诚实结论

- **稳定可交付的能力是真实的**:计数、相对时间、跨栈后端三件套等"有明确验证锚点"的部分,doubao-lite 在本系统编排下稳定产出正确代码并通过真实测试。
- **模型能力边界也是真实的**:纯视觉接线(把函数接进组件渲染)doubao-lite 做得不稳,这是模型层而非编排层的短板。
- **评测体系有牙**:独立审计抓出了测试与提交信息都没暴露的 reject 与系统级依赖污染,这比一份全绿报告更可信。

## 局限性

4 条为真实交付样本而非大规模基准集;审计员为模型而非人类终审,但"独立模型互审 + 对抗复核"为当前业界(Codex / Claude Code reviewer 机制)标准做法。
