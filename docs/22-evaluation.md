# 独立验收评测(方法与结果)

对应提交材料选填项「评测方案与样例结果」。回答一个核心质疑:**"模型自己写的测试通过"不等于"输出真的正确"**——生成系统既当运动员又当裁判时,需要独立的第三方审判。

## 评测方法

- **被评对象**:平台在 Conduit 实仓上真实交付的 4 条提测分支(纯前端统计、跨栈字段、列表打标、时间展示,覆盖 L1/L2 与新 Skill 现场注册案例)。
- **审计员独立性**:由与运行时模型(doubao-seed-2.0-lite)**完全无关**的另一模型体系担任,从未接触生成过程,指令明确"发现问题是你的价值,放水是失职"。
- **审需求而非审测试**:每个任务从原始需求拆出 5-7 条验收标准,审计员必须:
  1. 打开真实代码核对(不接受 commit message 自述);
  2. 对计数/时间类逻辑**人工推演具体输入的期望输出**并与实现对照;
  3. 在仓库目录独立运行 `npm test -- --run`;
  4. **独立判断测试断言本身是否正确**——专门防"实现错了,测试也跟着错"的共谋;
  5. 如实列出所有缺陷(需求遗漏/边界缺陷/风格突兀)。
- **对抗复核**:任何 FAIL 判定再派独立复核员审一遍,防误杀。
- **结论分级**:accept / accept-with-defects / reject。

## 局限性(如实声明)

- 4 条分支为真实交付样本,非统计意义上的大规模基准集;
- 审计员为模型而非人类专家终审,但"独立模型互审 + 对抗复核"为当前业界(Codex / Claude Code reviewer 机制)标准做法;
- 页面级行为(浏览器中真实渲染)由 visual gate 能力覆盖,数据依赖型页面需数据库,未纳入本轮静态审计范围。
