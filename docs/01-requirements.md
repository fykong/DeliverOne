# 产品需求整理

## 产品定位

这是一个本地全栈交付 Agent App，不是静态展示页，也不是只包一层模型聊天。用户应该能选择真实仓库、输入需求、看到 Agent 的真实输出、确认计划、让 Agent 在沙盒中修改代码、运行验证、查看预览、审查 diff，并在任意阶段回退。

## 第一版必须跑通的闭环

1. 用户选择本地仓库或公开 GitHub 仓库。
2. 系统为当前对话创建独立沙盒。
3. 用户输入真实需求。
4. Agent 读取仓库画像、memory 和 Skill，生成中文计划。
5. 用户确认计划。
6. 系统生成可审查的工具调用计划。
7. 用户确认工具计划后，系统按步骤调用受控工具。
8. 写入前创建 checkpoint，写入后记录 diff。
9. 运行验证命令并记录结果。
10. 右侧展示交付证据、checkpoint、diff、验证和预览状态。
11. 用户可以按 checkpoint 回退，也可以确认后全仓回退到沙盒原始 HEAD。

## UI 结构

页面使用三栏桌面布局。

左侧：历史对话、新建对话、仓库入口。

中间：Codex 风格对话区，只展示用户消息、Agent 消息和底部输入框。不要在中间堆系统说明、评委话术、无意义执行步骤、模型理解卡片。

右侧：交付检查器。放计划确认、工具调用证据、diff、checkpoint、验证、预览、回退和模型设置入口。不同功能用标签页或清晰分组，不混堆在主区域。

## 仓库和沙盒

- 支持本地路径。
- 支持公开 GitHub 仓库。
- 示例仓库为 `https://github.com/TonyMckes/conduit-realworld-example-app`。
- 每个对话都有独立沙盒。
- 默认不修改原始仓库。
- 只有用户明确授权时，才允许把沙盒变更应用回原始仓库。

## 模型

- 主模型为 Doubao / Ark。
- 模型配置独立放在 `config/model-providers.json` 和 `config/model-settings.json`。
- 模型切换写入磁盘。
- 已有对话不受切换影响，新对话使用新的默认模型。

## MCP 和 Skills

第一版先实现稳定的内置 MCP 风格工具层，再预留外部 MCP Server 接入。

当前内置工具层包括：

- 代码搜索、读取、diff、写入、补丁。
- 命令运行。
- 沙盒创建。
- checkpoint 和 rollback。
- 预览进程启动。

Skills 用来约束某类需求的流程，不是展示文案。新增需求模式应优先新增 Skill 文件，而不是改主编排代码。

## 执行模式

默认是计划模式：关键步骤需要用户确认。

自动交付模式第一版默认关闭，只在用户手动开启后使用。

无论哪种模式，写入前必须 checkpoint，高风险命令必须授权，工具调用必须记录事件。
