# 需求模式 Skill：文案与 UI 微改

## 适用场景

「把按钮上的『提交』改成『保存』」「修这个错别字」「占位符文案换一下」「标题文字改成 XX」——目标是字符串本身，不涉及结构、样式和逻辑。

## 流程

1. 用 clarifyChecklist 拿到逐字的旧文案和新文案；用户口述不全时，把搜索到的候选原文回显给用户确认，不要猜。
2. 逐字搜索定位，把全部命中点列表（文件、行、上下文）交用户圈定要改哪些。
3. 最小修改：只动字符串字面量或 i18n 资源值；包含旧文案的测试断言、快照同步更新并单独标注。
4. 双向对账：范围内旧文案搜索归零，新文案命中数与确认位置数一致。
5. 页面验证：browser.preview_smoke 用新文案做 expectedTexts 断言，以页面真实渲染作为验收证据，「构建通过」不算验收。

## 硬限制

- diff 中出现任何非字符串改动（标签、样式、逻辑、格式化）即为越界，回退后重做。
- 文字来源是后端或数据库时本模式不适用——停下来向用户说明，改走对应的后端模式。
- 不替用户润色：标点、大小写、空格一律以用户确认的版本为准。
- 共享 i18n key 或常量改值前，必须列出全部使用方并获用户确认。
