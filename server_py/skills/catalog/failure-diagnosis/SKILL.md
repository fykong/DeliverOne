# 执行失败诊断 Skill

## 触发场景

verification.run 红灯、command.run 非零退出、browser.preview_smoke 断言失败、构建或安装报错时使用。

## 四类归因（按排查顺序）

1. 环境类（最先排查，最常见）
   - 特征：错误发生在安装、导入或启动阶段，与本次改动的文件无关。
   - 签名：ENOENT、MODULE_NOT_FOUND、command not found、EADDRINUSE、engines 版本不满足、权限拒绝、网络超时。
   - 动作：装依赖、换端口、补路径；特别注意沙盒平台差异（Windows 路径分隔符、shell 语法、watch 模式挂起）。
2. 代码类
   - 特征：失败栈顶帧落在本次 diff 改过的文件或其直接调用方，且改动前基线是绿的。
   - 动作：对照 diff 缩小范围；多文件改动时按 checkpoint 单文件回退做二分定位。
3. 计划类
   - 特征：每个局部都「对」，整体却不对——改错了文件、漏改一处引用、步骤顺序错（先用后建）。
   - 签名：单测通过但验收断言不过；搜索命中的文件语义与需求对不上。
   - 动作：作废错误位置的改动，回定位阶段重搜，而不是在原地继续修。
4. 需求类
   - 特征：代码自洽、验证全绿，但行为与用户期望不一致；或实现依赖的字段、功能在仓库里根本不存在。
   - 动作：停止修改，带证据（现状截图、代码片段）回到澄清环节，给用户列出可选方案。

## 流程

1. 取证：失败四件套（命令、exit code、错误关键行、阶段）写入事件流。
2. 归类：按上述顺序判断，选最可能的一类，提出单一假设。
3. 验证假设：执行一个有预期的动作，复跑原失败命令。
4. 通过则跑 verification.run 防新增红灯；不通过则换下一个假设，本轮证据存档。
5. 三轮无进展，或四类全部排除，停止动手，产出 blocker 报告交用户决策。

## 硬限制

- 禁止不改变任何变量地重跑失败命令两次以上。
- 禁止为了「让验证通过」而删除或跳过失败测试，除非用户显式确认该测试本身有错。
- 每轮诊断必须产生新增证据（新文件、新命令输出、新日志），否则视为无进展计入三轮上限。
- blocked 不等于失败：证据充分的 blocker 报告优于盲目改代码。
