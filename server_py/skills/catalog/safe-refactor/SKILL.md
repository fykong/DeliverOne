# 需求模式 Skill：安全重构

## 适用场景

「把 X 抽成公共函数」「拆分这个大文件」「重命名这个模块」「删掉重复代码」「整理目录结构」——目标是结构变化、承诺行为不变的需求。

## 流程

1. 用 clarifyChecklist 钉死范围边界和行为承诺，把 nonGoals 显式写进计划。
2. 跑基线：verification.run 记录改动前的红绿状态，存量失败单独标注存档。
3. 画影响面：对每个要动的符号搜全引用（含字符串形式、配置、测试），引用点数量写进计划。
4. 小步循环：一个等价变换 → 复跑验证 → checkpoint → 下一步；任何一步变红，立即在该步内解决或回退该步，不带病前进。
5. 对账交付：diff 与基线逐文件核对确认零行为改动；过程中发现的 bug 单独列出上报，不混入本次交付。

## 硬限制

- 测试覆盖不足的核心路径，必须先补特征测试或拿到用户显式接受风险的记录，才能开始重构。
- 重构 diff 中出现行为性修改（新逻辑、新文案、依赖变更）即视为越界，拆出去单独走需求流程。
- 禁止一次写入跨越多个不可验证的中间态。
- 删除必须附零引用的搜索证据。
